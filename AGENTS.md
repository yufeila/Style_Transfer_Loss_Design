## 网络与代理

- 远程服务器可能无法直接访问外网。
- 克隆 GitHub、下载模型或数据时，优先通过本地代理建立 SSH 反向隧道。
- 使用代理前先确认本地代理协议和端口。
- 常见做法：

```bash
ssh -N -R <remote_port>:127.0.0.1:<local_proxy_port> gpu-server
```

- 然后在远程设置：

```bash
export HTTP_PROXY=http://127.0.0.1:<remote_port>
export HTTPS_PROXY=http://127.0.0.1:<remote_port>
export ALL_PROXY=http://127.0.0.1:<remote_port>
```

## 本地与远程同步规则

- 本地只保留核心代码、核心配置、核心结果图、最终报告材料和必要日志摘要。
- 大型数据集、模型权重、完整训练日志、大量中间图片和批量输出不复制回本地。
- 需要同步时，优先使用 `rsync -az`，并明确源目录和目标目录。
- 同步前先确认不会覆盖本地已有重要结果。

## 训练与实验策略

- 如果需要大规模训练，一开始就尝试最有希望的正式参数。
- 不要为了训练时间或模型能力，默认采用明显不合适的小模型、小分辨率或过弱参数。
- 若正式训练前需要测试流程能否启动，只允许做一次最小 sanity check。
- sanity check 只检查：
  - 环境能否导入依赖
  - 数据能否读取
  - GPU 能否使用
  - 模型能否前向和反向传播
  - 日志和输出能否正常写入
- sanity check 成功后，应直接转入正式实验配置。
- sanity check 成功后，应删除该检查产生的临时输出、临时日志和中间产物，避免污染正式实验记录。
- 不要连续做多个小型试探实验来替代正式推进。

## 长任务规则

- 长时间训练、批量生成或大规模评测应使用 `tmux`、`nohup` 或等价方式运行。
- 启动长任务后必须记录：
  - 运行命令
  - conda 环境
  - 代码版本或 commit
  - 配置文件
  - 日志路径
  - 输出路径
  - 预计检查点
  - 中止条件
- 给用户反馈时应同时给出监控命令。

## 可追溯记录

- 远程复现实验必须保留可追溯记录。
- 每次正式实验至少记录：
  - 仓库 commit 或当前代码状态
  - 运行命令
  - 关键配置
  - 环境信息
  - GPU 信息
  - 数据路径
  - 日志路径
  - 输出路径
  - 成功结果或失败原因
- 失败不是问题，但失败原因必须可追溯。

## 高成本与破坏性操作

- 高成本、破坏性或会长时间占用大量远程资源的操作前，必须先向用户确认。
- 例如：
  - 删除远程数据、权重或实验输出
  - 长时间独占 GPU
  - 下载大体量数据集或模型
  - 启动多组大规模并行实验
  - 覆盖已有核心结果
- 普通环境探查和一次最小 sanity check 可直接执行。
